# Changelog

All notable changes to this project will be documented in this file.

The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) and this project uses semantic versioning.

## [Unreleased]

### Added

#### Localization (i18n)
- Closed the `<Chorus labels>` localization gap for the newer feature surfaces that previously hard-coded English copy. `ChorusLabels` now carries four new sections — `cost` (cost-meter header label, "no usage" tooltip, budget suffix, per-message chip aria-label, live-estimate tooltip), `artifacts` (panel/card title fallback, version/diff/copy/download/open actions, react placeholder + error, per-kind labels), `approval` (approval card title, server prefix, input toggle, allow-once/allow-always/deny buttons), and `mcp` (connection-status line, error/reconnecting suffixes, reconnect button) — plus composer additions for the slash-command palette (`slashCommands`), MCP resource picker (`attachResource` / `resourcePickerPlaceholder`), model picker fallback (`modelPicker`), and the default tool loader's screen-reader "Calling …" status (`toolCall.calling`). All default to the existing English strings and merge with the same `null`/`undefined`/`''`-keep-default resilience as the other sections. Exported `ChorusCostLabels`, `ChorusArtifactLabels`, `ChorusApprovalLabels`, `ChorusMcpLabels` types and `DEFAULT_COST_LABELS` / `DEFAULT_ARTIFACT_LABELS` / `DEFAULT_APPROVAL_LABELS` / `DEFAULT_MCP_LABELS` constants. `<ChorusArtifactPanel>`, `<ArtifactCard>`, and `<ToolApprovalCard>` accept a `labels` prop for standalone (outside-`<Chorus>`) use, and the starter `CalendarPicker` block gained a `confirmLabel` prop (matching the existing `Form` `submitLabel` / `Image` `blockedLabel` targeted-prop pattern).

#### Roleplay / multi-character primitives
- Added an optional `MessageSpeaker` (`{ id, name, avatarUrl?, metadata? }`) on every message role. The default renderer uses `speaker.name` as the visible label above the bubble (falling back to the existing `ChorusSpeakerLabels` role defaults), the screen-reader announcement prefers `speaker.name` over the role label, and the Markdown export uses it for the per-message heading so a multi-character transcript reads `## Captain Hook` instead of `## Assistant`. Persistence round-trips the field automatically; built-in provider request mappers ignore it (use `transformRequest` or prepend to `text` if your provider needs the name on the wire).
- Added `<Chorus showSpeakerAvatars>` (and the same prop on `<ChatWindow>` / `<MessageBubble>`) which renders `message.speaker.avatarUrl` as a small circular `<img>` next to the visible name. The name itself renders unconditionally when `speaker` is present; only the avatar image is gated. The visible badge is `aria-hidden` so SR users hear the name once, not twice. Exported `MessageSpeakerBadge` and `resolveMessageSpeakerLabel` for custom shells.
- Added `<Chorus transformRequest>`, a pre-send hook fired immediately before each outbound `transport` request. The hook receives `{ messages, systemPrompt, signal, reason }` and may return `{ messages?, systemPrompt? }` overrides that apply to the wire request only (the persisted transcript stays clean). Fires once per turn for plain sends (`reason: 'initial'`) and once per iteration inside an `autoContinueTools` loop (`reason: 'tool-continuation'`). Throws route through the standard `onError` path; `AbortError` rejections are silent. Designed for lorebook injection, rolling summaries, RAG retrieval, and author's notes.
- Added `<Chorus conversationMetadata>` + `<Chorus onConversationMetadataChange>`, a controlled prop pair for persisting a small free-form object alongside the transcript at `${persistenceKey}::meta` in `persistenceStorage`. Designed for roleplay/multi-agent shells that need to persist active-character / persona / lorebook ids per conversation without pinning them onto every message. Pass `null` to clear the slot. Exported the underlying `useConversationMetadata(key, options)` hook (with `value` / `setValue` / `loaded` / `error` / `canPersist`) so hosts can wire conversation-scoped state outside `<Chorus>` too.
- Added the [`examples/with-roleplay`](./examples/with-roleplay) runnable example: two characters, a persona, a keyword-triggered inline lorebook injected via `transformRequest`, speaker avatars enabled, and the active-character slot persisted through `conversationMetadata`. No backend required.

#### Connectors / streaming
- Added the `'ai-sdk'` connector for the [Vercel AI SDK](./docs/guide.md#vercel-ai-sdk-stream-format). It parses both stream shapes the SDK can emit: the SSE-formatted **UI message stream** (`toUIMessageStreamResponse()`, AI SDK v5+) with `text-delta`, `reasoning-delta`, and `tool-input-*` / `tool-output-*` JSON events, and the prefix-coded **data-stream protocol** (`toDataStreamResponse()`, AI SDK v4) with `0:` / `g:` / `9:` / `c:` / `a:` text, reasoning, and tool frames plus `d:`/`e:` finish frames. `error` / `3:` frames surface through the in-band error path, and protocol/lifecycle frames are silently ignored.
- Extended `autoConnector` frame detection to recognize Vercel AI SDK UI-message-stream JSON and data-stream prefix lines, so AI SDK routes work under the default `connector="auto"`.
- Exported `createOpenAIConnector`, a factory for OpenAI connectors with a configurable, case-insensitive `<think>` reasoning splitter (`thinkTag` option), along with the `OpenAIConnectorOptions` and `ThinkTagSplitterOptions` types.
- The new `'ai-sdk'` connector is reachable by name only — `getConnector('ai-sdk')` or `connector="ai-sdk"`. The public connector accessors remain `getConnector` and `createOpenAIConnector`; the provider singletons (`openaiConnector`/`anthropicConnector`/`geminiConnector`/`aiSdkConnector`) and `autoConnector` stay `@internal` and are not exported from `react-chorus` / `react-chorus/headless`.

#### Developer experience
- `<ChatInput>` now emits a one-time `console.warn` in development when a host passes `onKeyDown`. As documented on `ChatInputProps`, an `onKeyDown` spread via `...rest` attaches to the composer container `<div>`, not the inner `<textarea>`, so it never observes Enter (the composer `preventDefault()`s it) or other textarea-consumed keys — a host wiring slash-command or up-arrow-to-edit shortcuts would otherwise get silent partial behavior with no diagnostic.
- `<Chorus>` now emits a one-time `console.warn` in development when `showCost` is set with `onSend` and no `transport`. The cost meter reads `metadata.usage` written by the transport-path connectors' `onStreamMetadata` wrapper, which never runs on the `onSend` path — so the chips stayed at `$0` with no signal that the prop was inert. The warning points at the new `onSend`-path recipe (hand-roll `metadata.usage` on each finalized assistant message) in `docs/api.md`.
- The built-in `Image` block now accepts `allowedProtocols?: string[]` (default `['https:', 'data:image/']`) and `blockedLabel?: string`. Hosts that need to opt into additional URL schemes (e.g. `http://localhost` for dev-server screenshots) or relocalize the placeholder can wrap the block so the host-supplied list is pinned AFTER the model props are spread — the model output remains untrusted and cannot widen the whitelist by passing its own `allowedProtocols`. Documented in the new `react-chorus/blocks` section of `docs/api.md`.

#### Documentation
- New `docs/api.md` "Subpath exports" section documents the five tree-shakeable subpaths shipped alongside the root barrel: `react-chorus/blocks`, `react-chorus/blocks/Chart`, `react-chorus/loaders`, `react-chorus/validators`, and `react-chorus/pricing`. Each subpath gets a per-section import recipe, named-export list, and pairing with the matching `<Chorus>` prop. A new "Cost meter" section covers `showCost` / `pricing` / `budgetAlert` end-to-end, including the `onSend` recipe for hand-rolling `metadata.usage` when the transport-path connector's automatic wiring is not available. The README's Generative-UI blocks bullet now cross-links the new docs and the runnable [`examples/with-blocks`](./examples/with-blocks) demo.

### Changed

#### Markdown
- The bundled `chorus-md` stylesheet now styles GFM tables. `marked`'s `gfm: true` already emitted real `<table>` markup, but the injected sheet only styled code blocks, so tables rendered as borderless rows of text; they now use collapsed borders, 1px cell borders, cell padding, and an emphasized header row. The new `--chorus-md-table-border` and `--chorus-md-table-header-bg` CSS variables theme the cell borders and header background. This is a visible rendering change for every `<Markdown>`/`<Chorus>` consumer that renders tables.

#### Documentation & tooling
- Tightened the package `engines.node` range from `>=20.19.0` to `>=20.19.0 <21 || >=22.12` — and the matching `verify:examples` `REQUIRED_NODE_RANGE` constant and every example `package.json` — so it states the real Vite 8 toolchain requirement. Installing on Node 21.x or 22.0–22.11 now emits an `npm WARN EBADENGINE` warning instead of silently permitting a version the toolchain rejects.
- Restructured the README into onboarding-only content (what it is, install, Quick start, the two send paths, examples). The full reference moved into a new `docs/` directory — [`docs/guide.md`](./docs/guide.md) (usage, connectors, stream formats), [`docs/api.md`](./docs/api.md) (props, hooks, components, tools, theming), [`docs/deployment.md`](./docs/deployment.md) (bundle size, SSR, CSP), and [`docs/migration.md`](./docs/migration.md) — plus a new [`docs/uploads.md`](./docs/uploads.md) end-to-end `uploadAttachment` recipe for large or non-image attachments.

### Fixed
- The default message deserializer now validates `createdAt`. A stored entry whose `createdAt` is present but not a string (e.g. a number or object) is dropped with the same dev warning as other invalid entries, instead of reaching the timestamp renderer where it formatted to `Invalid Date` under `<Chorus showTimestamps>`. This matches the codec JSDoc, which already claimed each entry is validated against the public `Message` contract.
- `createRandomId` now applies its `prefix` on the `crypto.randomUUID` path too. Previously it returned a bare, prefix-less UUID in modern browsers and a prefixed string only in WebCrypto-less runtimes/tests, so default conversation ids — used as storage-key suffixes — were inconsistently shaped across environments. Generated ids are now `${prefix}-…` on both code paths.
- Deleting the user message tied to a stream error now dismisses the error banner and disarms Retry. Previously `handleDelete` left `streamError`/`streamRawError` and `lastSubmittedTurnRef` untouched, so after a transport stream error the banner outlived the deleted message and clicking Retry replayed the stale submitted turn — re-inserting the just-deleted user message into the transcript. `handleDelete` now clears the stream error and submitted-turn ref whenever it removes the last user turn (or any message the armed Retry would replay).
- The built-in transcript error banner now renders a dismiss (X) button when `onDismissError` is wired (it always is under `<Chorus>`), so consumers using the default error UI — not just a custom `renderError` — can clear the banner. Added a localizable `labels.transcript.dismissError` string for its accessible label.
- `ChorusRef.dismissError()` is no longer rejected while `disabled`/`readOnly` or a built-in persistence load is pending. Dismissing a stream error mutates only transient error state — not the transcript — and the built-in banner's dismiss button was already available in those modes, so the imperative API now matches it. (`dismissError()` is still rejected in controlled mode with no `onChange`.)
- `ChatWindow`/`Chorus` `maxRenderedMessages` windowing no longer breaks streaming state for windowed-out turns. The in-flight tool-call turn is now derived from the full visible array before windowing, so a small `maxRenderedMessages` that slices the last user message out can no longer flip older finished tool calls to "Running…". The message tagged by `streamingMessageId` is also force-included in the rendered window, so its partial text keeps rendering (in streaming-plain-text mode) even when host-appended rows or a very low `maxRenderedMessages` would otherwise push it out of the trailing slice.
- The exported `MessageBubble` / `MessageRow` now render a `ToolCallBlock` for `role: 'tool'` messages (with `message.text`, when present, shown above as a host-authored summary), so a host composing a custom shell with the exported components gets the structured tool call instead of an empty bubble — previously only `ChatWindow`'s tool branch rendered one. Added an optional `toolCallLabels` prop on `MessageRow` / `MessageBubble` / `MessageBubbleLayout` to relabel that block.
- The streaming reasoning `<details>` is no longer force-controlled open. `MessageReasoning` now tracks the reader's own toggle, so collapsing the chain-of-thought mid-stream sticks instead of being re-forced open on every streamed chunk; the transcript's auto-open for a reasoning-only streaming turn is now only a starting suggestion the reader's choice overrides.
- An auto-opened streaming reasoning `<details>` no longer collapses out from under the reader when the first answer token arrives. `MessageReasoning` now latches the auto-open hint: once it has been `true` the disclosure stays open through the answer (the hint clears the instant answer text appears) until the reader explicitly collapses it, instead of snapping shut on its own while the reader is still following the chain-of-thought.
- A persistence-backed `<Chorus>` that unmounts mid-stream again flushes its final RAF-buffered assistant token into persistence, so the stored transcript no longer ends mid-token. A real unmount now only aborts the in-flight controller and leaves the partial turn to `useRAFQueue`'s persist-only unmount flush — it no longer rewrites the transcript through `onChange` / `onMessagesChange` after teardown — while a `persistenceKey` switch still fully supersedes the stale turn.
- A corrupt cross-tab `storage` event no longer drops a pending debounced local write (lost update). `localStorageSync`/`crossTabSync` previously discarded an armed-but-not-started debounced write *before* trying to apply the external value, on the assumption it would supersede the dropped write. When the external payload was corrupt — written by another tab, a browser extension, an older library version, or a partial `QuotaExceededError` write — `stateFromRaw` rejected it and returned without applying, so the dropped write (a debounced message write of coalesced streamed tokens, or a `touchConversation` `updatedAt` recency bump) was lost and a reload silently discarded the change. The armed debounced write is now dropped only once the external value is confirmed applicable (after a successful parse, before `setState`), so a rejected corrupt event leaves the timer to fire and persist as intended.
- `ChorusRef.scrollToMessage(id)` no longer fails silently for a valid message that is not rendered. When `id` is a known message (one `getMessages()` returns) whose row is absent from the DOM — windowed out by `maxRenderedMessages`, hidden by `hiddenRoles`, or drawn by a custom `renderMessage` that did not spread `ctx.messageProps` — it now logs a one-time development warning so the `false` is distinguishable from a `false` for an unknown id. The `ChorusRef.scrollToMessage` JSDoc and README now document the windowing limitation and the `getMessages()` cross-check that tells the two cases apart.
- An `onSend` that both streams via the helpers (`appendAssistant`/`appendReasoning`/`appendToolDelta`) **and** returns a `Message` now emits a one-time dev warning. The `ChorusOnSend` contract is "do exactly one", and the streamed output already wins, but the dropped return value was previously silent — unlike the empty-`onSend` and auto-finalize misuses, which both warn.
- Clarified the `onChunk` contract: its `ChorusProps` JSDoc (and the README props table) now state explicitly that `onChunk` reports streamed assistant **text** only — reasoning/thinking deltas and tool-call deltas never trigger it — so a host using `onChunk` to mirror or measure streamed output knows reasoning tokens are excluded.
- The dev-mode ignored-prop warnings now use `isTransportPresent` instead of `transport === undefined` / bare truthiness, matching the send pipeline's transport-vs-`onSend` branching. The `connector` / `connectorOptions` / `onStreamDone` / tool-prop warnings now fire for `transport={null}` + `onSend` (where the real send path is `onSend`), and the "`sending` alongside `transport`" warning now fires for a misconfigured `transport=""` (which still reports transport-owned busy state).
- The `ai-sdk` connector now tags streamed tool deltas with `provider: 'ai-sdk'`, so the captured provider tool-call id is persisted into a provider-neutral `metadata.aiSdk.toolCallId` slot. Previously the connector set `providerId` but no `provider`, so `metadataWithToolProvider` skipped both its `openai`/`anthropic` branches and silently dropped the id — a tool-call round-trip through `connector="ai-sdk"` lost the tool-call id that `autoContinueTools` needs to re-attach when replaying the result.
- The `ai-sdk` connector now surfaces token usage and finish reason from its terminal frames. The UI-message-stream `finish` / `finish-message` frames and the data-stream `d:` finish-message frame run their `usage` through the shared `extractUsage` normalizer (extended to recognize the AI SDK's own `promptTokens`/`completionTokens` and `inputTokens`/`outputTokens` field names) and attach `{ metadata: { usage, finishReason } }` alongside `done: true`, so a cost/usage telemetry consumer wiring `onMetadata` / `onStreamMetadata` now sees usage for `connector="ai-sdk"` (and AI-SDK streams auto-detected by `connector="auto"`) like every other connector.
- The OpenAI Chat Completions connector no longer emits `metadata.usage` once per chunk. A proxy that attaches a cumulative `usage` object to every streamed chunk previously fired `onMetadata` on each one, over-counting any non-idempotent consumer (e.g. a running cost counter). The connector now buffers the latest `usage` and surfaces it exactly once, on the terminating chunk (`finish_reason` present or the trailing `choices: []` frame) — matching the Gemini and Responses connectors. The Gemini connector gained a parity guard so an atypical `{ candidates: [], usageMetadata }` frame surfaces its usage instead of being dropped.
- The provider-request mappers no longer drop or mis-serialize a handler-less, record-shaped `tools` registry. A `Record<string, ChorusToolDefinition>` whose values all omit `handler` (the documented server-side-execution escape hatch) previously serialized to an empty `tools` array — the model was never told the tools existed — and, since `isChorusToolsSource` did not recognize it as Chorus-shaped, was forwarded as a bare record object on `tools` (every provider rejects a non-array `tools` with a 400). `toToolDefinitionList` now keeps every non-function record entry and `isChorusToolsSource` recognizes handler-less definition records, so all four mappers serialize them exactly like a definition array.
- Three provider-request mapper divergences are fixed so OpenAI Chat Completions, OpenAI Responses, Anthropic, and Gemini serialize the same input identically: (1) an explicit empty-string `toolCall.output` is now honored in the OpenAI Chat/Responses and Anthropic tool results instead of falling through to the streamed message text (Gemini already honored it); (2) a tool call with an empty-string `toolCall.name` keeps its structured form on Gemini (named `tool`) instead of degrading to a prose fallback, matching the other three mappers; (3) history `role: 'system'` text is trimmed before it reaches the Anthropic `system` string and the Gemini `systemInstruction`, matching the Chat Completions and content-part mappers, which already trim.

### Deprecation candidates (future major)
- The default transport body `{ prompt, history }` duplicates the latest user turn — `prompt` equals `history[history.length - 1].text`. Backends already consume `history` only (see all `examples/` proxies). A future major release should drop `prompt` from `createFetchSSETransport`, `createWebSocketTransport`, and `createDefaultFetchSSETransport` defaults and send `{ history }` exclusively. Until then, the docs and JSDoc warn against re-appending `prompt` server-side. See [Migration and Upgrading → Default transport body will drop the `prompt` field](./docs/migration.md#default-transport-body-will-drop-the-prompt-field) for the concrete migration path.

## [0.2.0] - 2026-05-15

### Added

#### Tool agent loop
- Added a `tools` registry prop and `ChorusToolRegistry` type for declaring client-side tool implementations.
- Added `autoContinueTools`, `maxToolIterations`, and `shouldContinueToolLoop` props for controlling automatic tool-call → result → resume cycles.
- Added `continueOnToolError`: when a tool handler (or `onToolCall`) throws a non-abort error, the error is recorded on the tool row and — with `autoContinueTools` enabled — fed back to the model as the tool result so the loop self-recovers instead of ending the turn with the error banner.
- Added `onToolCall`, `onToolDelta`, `onStreamDone`, and `onAbort` callbacks with `ChorusToolCallContext`, `ChorusToolDeltaContext`, `ChorusToolLoopContext`, `ChorusStreamDoneContext`, and `ChorusAbortContext` payload types.

#### Multi-conversation
- Added the `useConversations` hook and `<ConversationList>` component for listing, switching, renaming, and deleting saved conversations.
- Added `ConversationSummary`, `UseConversationsOptions`, `UseConversationsResult`, `ConversationStorageError`, `ConversationStorageOperation`, and `RenameFromFirstMessageOptions` types.
- Added `confirmDeleteConversation` (with `ConfirmDeleteConversation`/`ConfirmDeleteConversationContext`) for custom delete-confirmation UX.

#### Composer / UX
- Added `disabled`, `disabledReason`, and `readOnly` props for locking the composer and disabling message actions.
- Added `resetToInitialMessages` for restoring the seeded message list and `confirmDeleteMessage` (with `ChorusConfirmDeleteMessage`/`ChorusDeleteMessageContext`) for delete-confirmation UX.
- Added `suggestedPrompts`, `showJumpToBottomButton`, and `systemPrompt` props.
- Added `getMessageFeedback`/`onFeedback` (with `GetMessageFeedback`/`MessageFeedback` types) for per-message thumbs-up/down feedback.
- Added `maxRenderedMessages`, `renderError`, `renderMessage`, `markdownProps`, and `markdownSanitizer` props with `RenderErrorContext`, `RenderMessageContext`, `RenderMessageRootProps`, `MessageMarkdownProps`, and `MessageRenderActions` types.

#### Persistence
- Added `onPersistenceError`, `deserializeMessages`, `serializeMessages`, and `PersistenceWriteOptions` for richer custom storage adapters.
- Added `flush` on `useChorusPersistence` for forcing pending writes (with `ChorusPersistenceError`, `PersistenceOperation`, `DeserializeMessages`, `SerializeMessages`, `UseChorusPersistenceOptions`, `UseChorusPersistenceResult` types).
- `StorageAdapter` now supports async loading; pending loads are awaited before send/persist.

#### Provider helpers (new `react-chorus/provider-requests` subpath)
- Added `formatOpenAIChatCompletionsBody`, `formatOpenAIResponsesBody`, and `toOpenAIResponsesInput` for OpenAI request bodies.
- Added `formatAnthropicMessagesBody`, `toAnthropicMessagesBody`, and `toAnthropicMessages` for Anthropic Messages request bodies.
- Added `formatGeminiGenerateContentBody`, `toGeminiContents`, and `toGeminiGenerateContentBody` for Gemini GenerateContent request bodies.
- Added `ProviderMappingOptions` and `UnsupportedAttachmentText` for cross-provider mapping config, plus per-provider body/option/message types (`OpenAIChatCompletionsBody`/`Options`/`Message`, `OpenAIResponsesBody`/`Options`/`InputItem`, `AnthropicMessage`/`MessagesBody`/`MessagesBodyOptions`, `GeminiContent`/`GenerateContentBody`/`GenerateContentBodyOptions`).
- Exported the same helpers from the root entry, with the dedicated `react-chorus/provider-requests` subpath available for consumers who want to import only the mappers.

#### Transports (new `react-chorus/transport` subpath)
- Added `createWebSocketTransport` with persistent-WebSocket mode and `WebSocketTransport`/`WebSocketTransportOptions` types.
- Added `onOpen`/`onClose`/`onError`/`onMessage` lifecycle callbacks and `formatBody`/`formatMessage` options on the SSE and WS transports.
- Added the `react-chorus/transport` subpath re-exporting `createFetchSSETransport`, `createWebSocketTransport`, and the `Transport` type for transport-only consumers.

#### Connectors / streaming
- Made `Connector<State>` generic with optional `createState`/`flush` hooks so connectors can carry per-stream state.
- Added in-band error parsing across connectors so provider error frames surface as `ChorusStreamError` instead of silently ending the stream.
- Added richer OpenAI Responses event coverage and Gemini blocked-finish-reason errors.
- Added `<think>` tag splitting so chain-of-thought output renders separately from the visible answer.
- Added `provider`, `providerId`, and `generated` fields to `ConnectorToolDelta` so connectors can report provider-issued tool-call IDs.
- Exported `getConnector` and `createOpenAIConnector` as the connector public API, plus the `Connector`/`ConnectorResult`/`ConnectorToolDelta`/`ConnectorWarning`/`OpenAIConnectorOptions`/`ThinkTagSplitterOptions` types.
- Added a `connectorOptions` prop on `<Chorus>` and option on `useChorusStream`/`useAssistantSession` that forwards connector customization (e.g. a custom `thinkTag` delimiter pair) to the built-in `'openai'` connector resolved from a `connector` string. Previously `getConnector`'s `options` argument was unreachable from the widget and hook.

#### Markdown
- Exported the standalone `Markdown` component with `MarkdownProps` and `MarkdownSanitizer` types.
- Added configurable `markedOptions`, `markedExtensions`, and sanitizer override for the Markdown pipeline.

#### Theming
- Added a `palette` prop to `<ChatWindow>` and `<ChatInput>` so every exported root component (`Chorus`, `ChatWindow`, `ChatInput`, `ConversationList`) themes through the same `--chorus-*` CSS-variable mechanism — making `<ChorusTheme>` an interchangeable wrapper form rather than the only theming path for composed shells.
- Documented a single theming entry point and the per-variable CSS-cascade precedence between `<Chorus palette>`, `<ChorusTheme>`, component `palette` props, and host CSS variables.

### Changed
- Bumped the package to `0.2.0` for a public message typing refinement.
- Built-in message feedback can now be cleared: clicking the already-active thumb toggles the rating off instead of being ignored. `onFeedback` (and the `MessageRenderActions.feedback` action) now receive `'up' | 'down' | null`, where `null` reports the cleared state.
- Settled on a single connector public API: built-in connectors are selected by name through `getConnector` (and the `connector` prop/option). The provider connector singletons (`openaiConnector` / `anthropicConnector` / `geminiConnector` / `aiSdkConnector`) and `autoConnector` are now `@internal` and are no longer re-exported from `react-chorus` / `react-chorus/headless`. See [Migration and Upgrading → Connector public API](./docs/migration.md#connector-public-api-getconnector-is-canonical).
- Replaced the public `Message` shape with a discriminated union (`AnyChorusMessage`) so `role: 'tool'` requires `toolCall`, non-tool messages forbid it, and tool/system messages reject attachments.
- Extracted Chorus send/session orchestration into `useAssistantSession` and clarified controlled, transport, connector, and sending-state development warnings.
- `useChorusStream.send()` now rejects non-abort stream failures after cleanup so `onSend` bridges can surface Chorus errors.
- `useChorusStream.send()` now rejects with a `ChorusStreamError` whose `code === 'concurrent-send'` when called while a previous send is still in flight, instead of silently resolving with `undefined`. The transport is still not invoked a second time and the dev-mode warning is preserved; custom shells that `await send(...)` can now distinguish the re-entrant no-op from a successful empty stream.
- Documented keyboard shortcuts, standalone `useChorusStream`, persistence examples, error handling, and OpenAI proxy buffering headers.
- `<ConversationList palette>` now applies the palette in `headless` renders too, matching `<Chorus>`/`<ChatWindow>`/`<ChatInput>`; a host-supplied palette is treated as a theme rather than default styling.

### Removed
- Removed the `openaiConnector`, `anthropicConnector`, `geminiConnector`, `aiSdkConnector`, and `autoConnector` named exports from the `react-chorus` and `react-chorus/headless` barrels. The provider connector singletons and `autoConnector` are now `@internal` (see "Changed" above) — this is an export-surface break, not just an internal-tag change: pre-0.2.0 code importing them directly (e.g. `import { openaiConnector } from 'react-chorus'`) no longer resolves. Select a built-in connector by name through `getConnector` or the `connector` prop/option instead.

### Fixed
- Fixed stream cleanup on unmount/pre-aborted signals, richer HTTP error details, WebSocket close-before-open hangs, safe dev-mode checks without `process`, observer callback isolation, and transport concurrency guards.
- Fixed the object-form `transport` shorthand (`FetchTransportInit`) ignoring `method`: it now accepts the same `'GET' | 'HEAD' | 'POST' | 'PUT' | 'PATCH' | 'DELETE'` set as `createFetchSSETransport`, including body-less `GET`/`HEAD` requests that skip `formatBody` and the default JSON `Content-Type`, instead of always issuing a `POST` against the configured `url`.
- A misconfigured `transport` no longer fails silently: an empty/whitespace-only URL string, or a transport object without a usable string `url` (missing, `undefined`, empty, or whitespace-only), now warns in development and resolves to a transport that rejects with a descriptive error so the existing stream-error UI surfaces it, instead of ending the assistant turn with a blank message or POSTing to the current document URL. A genuinely absent `transport` is unaffected.

## [0.1.0] - 2026-05-11

### Added
- Initial public release: the batteries-included `<Chorus>` chat component coordinating message rendering, the composer, streaming, persistence, and theming, with a stable `ChorusProps` contract.
- `react-chorus/headless` entry and the `headless` prop for unstyled, bring-your-own-CSS usage.
- Component pieces for custom shells: `ChatWindow`, `MessageBubble`, `ChatInput`, `ToolCallBlock`, `ChorusTheme`, and `Markdown`.
- The `Message` type with `Role`, `Attachment`, `ToolCall`, and `StorageAdapter`, including an optional generic `metadata` field and `system` / `tool` message roles.
- SSE streaming via the `transport` prop, with the `createFetchSSETransport` and `createWebSocketTransport` factories and the standalone `useChorusStream` hook.
- Provider connectors with auto-detection: `getConnector`, `autoConnector`, `openaiConnector`, and `anthropicConnector` for parsing OpenAI and Anthropic (Claude) SSE streams.
- Markdown rendering with fenced code blocks, lazy-loaded `highlight.js` syntax highlighting, and copy-to-clipboard buttons.
- Conversation persistence through the `useChorusPersistence` hook and `persistenceKey` prop, backed by `localStorage` or a pluggable `StorageAdapter`.
- Error-state UI with retry support.
- Message actions: edit a sent message, regenerate a response, and delete messages.
- File and image attachment support in `ChatInput`.
- Tool call / agent step rendering via `ToolCallBlock`, plus the `renderMessage` render prop and `MessageBubble` for custom message rendering.
- Theming through `ChorusTheme` and the `Palette` type.

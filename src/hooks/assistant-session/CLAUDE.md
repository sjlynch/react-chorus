# assistant-session internals

- Keep `useAssistantSession` and the `UseAssistantSessionResult` shape exported from `../useAssistantSession.ts`. Public context/handler types live in `types.ts` and are re-exported from the facade so consumers never import from this folder directly.
- `types.ts` — shared public context/handler types (`ChorusSendHelpers`, `ChorusFinishContext`, `ChorusAbortContext`, `ChorusStreamDoneContext`, etc.) plus the internal `UpdateMessagesOptions` / `UpdateSessionMessages` / `SubmittedUserTurn` aliases used by sibling modules.
- `messageUtils.ts` holds pure message/id/normalization helpers only.
- `sessionState.ts` and `sessionBusy.ts` collect facade-local state/busy setup so `useAssistantSession.ts` stays a top-level assembly of refs, buffer, orchestrator, tools, transport lifecycle, and commands.
- `transportResolver.ts` owns the `transport` prop resolution rules, including the absent-transport empty-200 fallback and the misconfigured-URL rejecting transport.
- `transport.ts` is the string-URL / `FetchTransportInit` object shortcut for the `<Chorus transport=...>` prop. It deliberately duplicates the tiny default fetch request shape instead of importing the public transport chunk, preserving the transport subpath budget; keep it aligned with `createFetchSSETransport` when the default body or option surface changes.
- `toolLoop.ts` owns the automatic tool-loop cap rules; `Infinity` means unlimited and invalid values warn once in dev.
- `observer.ts` contains the guarded callback warning helper; observer failures must not interrupt rendering.
- `observerCallbacks.ts` — `createObserverCallbacks` factory that wraps each host-supplied callback ref in a `try/catch` that forwards failures to `warnObserverError`. The factory runs once per session; the returned wrappers read the latest callback from each ref on every call.
- `assistantBuffer.ts` — `useAssistantBuffer` owns the RAF-buffered text/reasoning queues, the `pendingAssistantId` / `pendingToolMessageIds` / `toolMessageIdsByDeltaId` / `hasStartedAssistant` refs, and the `startAssistant` / `appendAssistantNow` / `appendAssistantReasoningNow` / `finalizeAssistantNow` / `resetStreamState` / `resetPendingAssistantState` mutators. `resetPendingAssistantState()` is the shared 6-line reset used by `completeActiveSession`'s forced-message branch and `finishTransportStream`'s catch arm. The `useRAFQueue` unmount flush hands the final buffered token to `updateSessionMessages` with `persistOnly: true`, so a trailing chunk still reaches persistence but never invokes a controlled host's `onChange` / the uncontrolled `setInternalMsgs` / the `onMessagesChange` observer after `<Chorus>` has torn down.
- `toolExecution.ts` — `useToolExecution` wires refs/callbacks around the tool message mutators (`toolMessageIdForDelta`, `appendToolDeltaNow`, `setToolOutput`, `setToolErrorOutput`), `createToolCallContext`, and the `runCompletedToolCalls` loop. Keep pure transforms (`buildToolMessageFromDelta`, `applyToolOutput`, `createToolCallContextFromMessage`) here so delta metadata/provider ids, output/error flags, and context shape stay centralized. `resolveToolHandlerLocal` is co-located here and intentionally re-implemented locally so the assistant-session chunk never imports `src/tools.ts`; the comment on the function spells out the chunk-isolation reason. Other modules in this folder must not import `resolveToolHandlerLocal`.
- `sessionHelpers.ts` — `createSessionHelpers` is a plain factory (no React) that takes a dep bag plus `(sessionId, signal, startedAt)` and returns the `ChorusSendHelpers` exposed to `onSend`, including the `minAssistantDelayMs` buffering and auto-finalize logic.
- `onSendLifecycle.ts` — `startOnSendLifecycle` owns the selected custom `onSend` branch: abort-controller setup, `createSessionHelpers` wiring, returned-message normalization, auto-finalize warnings, non-abort error reporting, and controller cleanup.
- `transportLifecycle.ts` — `useTransportLifecycle` owns `historyForTransport`, `startTransportStream`, `decideToolLoopContinuation`, and the internal `finishTransportStream` that runs queued tool calls, emits `onStreamDone`, and either continues the loop or releases transport busy state. Keep pure history/tool-loop/context/finalization helpers in `transportHistory.ts`, `transportToolLoop.ts`, `transportStreamDone.ts`, and `transportFinalizers.ts`. The `SendCallbacks` bag passed to `doStream` is built by `createTransportSendCallbacks` (see below) so `startTransportStream` reads as start/finalize-on-done/finalize-on-error control flow rather than a wall of guarded passthroughs.
- `transportSendCallbacks.ts` — `createTransportSendCallbacks` factory that returns the `SendCallbacks` object passed to `doStream`. It wraps each delegating callback (`onChunk` / `onReasoning` / `onSource` / `onToolDelta` / `onWarning` / `onMetadata`) in an `isAssistantSessionActive(sessionId)` guard so stale streams that outlive their session are dropped silently; `onDone` and `onError` are passed through as-is so the caller keeps the control-flow shape visible at the call site.
- `sessionCommands.ts` — `useSessionCommands` assembles the public `send` / `retry` / `stop` / `clear` / `handleEdit` / `handleRegenerate` / `handleDelete` callbacks from focused command hooks. Keep command-specific pure transforms in `sessionCommandTransforms.ts` and shared async confirmation semantics in `confirmationFlow.ts`.
- `sessionPath.ts` — pure send-path helpers (`selectAssistantSendPath`, `resolveAbortSendPath`) used by the orchestrator so transport presence keeps using `isTransportPresent` rather than truthiness (notably `transport=""`).
- `sessionWarnings.ts` — warning-once hook and exact dev warning strings for missing response handlers, ignored `onSend` when `transport` is present, and empty `onSend` turns.
- `sessionOrchestrator.ts` — `useSessionOrchestrator` owns the `activeSessionId` / `activeSendPath` / `controllerRef` bookkeeping and exposes `beginAssistantSession` / `isAssistantSessionActive` / `invalidateAssistantSession` / `completeActiveSession` / `removePendingAssistant` / `abortActiveAssistant` / `triggerAssistant` / `warnMissingResponseHandler` as a single hook. Dev-mode warning callbacks come from `sessionWarnings.ts` so each fires at most once per hook instance; `warnEmptyOnSend` is passed into `startOnSendLifecycle`. A cleanup effect aborts the in-flight controller on unmount AND on `persistenceKey` change (conversation switch) — it depends only on `persistenceKey` and reads `abortActiveAssistant` through a ref so an unstable `flushPersistence` cannot churn it into aborting a healthy turn. Two hook-supplied callbacks (`appendToolDeltaNow`, `startTransportStream`) are late-bound via `bindLateDeps` because the orchestrator is created before `useToolExecution` / `useTransportLifecycle` (those hooks consume `isAssistantSessionActive` / `invalidateAssistantSession` / `removePendingAssistant` from the orchestrator). On the transport path `abortActiveAssistant` cancels the stream by aborting its own `controllerRef` (which IS the external signal passed to `useChorusStream.send`); it does not call `useChorusStream.abort()`, which would log a spurious externalSignal dev warning. `controllerRef` is shared with `useTransportLifecycle` via the orchestrator's return value so both modules see the same in-flight controller.
- `useAssistantSessionRefs.ts` — `useAssistantSessionRefs` bundles the 25 `useLatestRef` wirings the facade needs (latest values of props/callbacks) into one helper call so the orchestration logic in `useAssistantSession.ts` isn't buried under ref boilerplate. It's a one-screen mechanical helper, not a new abstraction layer — when adding a new prop that needs latest-ref access, extend this helper rather than re-introducing per-ref lines in the facade.
- `assemble.ts` — `useAssistantSessionAssembly({ options, refs })` composes the sub-hooks (`useAssistantSessionState`, `createObserverCallbacks`, `useAssistantBuffer`, `useSessionOrchestrator`, `useToolExecution`, `useResolvedAssistantSessionTransport`, `useChorusStream`, `useSessionBusy`, `useTransportLifecycle`, `useSessionCommands`) and returns the public `UseAssistantSessionResult`. The facade in `../useAssistantSession.ts` is a thin shell: destructure options → `useAssistantSessionRefs` → `useAssistantSessionAssembly` → return, mirroring the `useChorusShellRuntime` pattern Chorus.tsx already uses. The `bindLateDeps` ordering between the orchestrator and `useToolExecution` / `useTransportLifecycle` and the unmount/`persistenceKey`-change cleanup live inside this assembly, not in the facade.

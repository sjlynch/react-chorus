# Persistence hook internals

- `messageCodec.ts` owns JSON defaults plus the default `Message` validator. Custom `deserializeMessages` callbacks intentionally receive only an array guard after they return; callers that opt in to custom decoding own any deeper validation/revival rules.
- `writeQueueCore.ts` is the generic serialized/debounced write queue (`useWriteQueueCore`) shared by message persistence and conversation-index persistence (`conversations/indexWriteQueue.ts`). It owns the pending-write ref, debounce timer, async write chain, in-flight tracking, the page-lifecycle fast path (`flushForPageLifecycle`/`runWriteImmediately`), and the `writeCoordination` (`isWritePending` / `hasPendingWrite` / `whenWriteSettles` / `dropPendingWrite`) cross-tab lost-update guard. `isWritePending` covers a write that is *executing*; `hasPendingWrite` covers a debounced write whose `setTimeout` is *armed but not started*. A cross-tab event must handle both: defer behind an in-flight write, and `dropPendingWrite()` an armed one so its stale snapshot cannot fire after the external value lands. `runWriteImmediately` must chain onto `writeChainRef` (never replace it) when a write is still in flight, or the prior write is orphaned and a later flush can interleave with it. The `deferSyncSettle` option decides whether a *synchronous* write stays write-pending for one microtask: message persistence passes `true` (so a same-task `pagehide` flush chains behind a just-flushed write); the conversation index passes `false` (so a cross-tab event right after a local write applies at once). Owning hooks supply `performWrite`/`reportWriteError`/`onWriteSuccess` and the opaque `TWrite` payload.
- `writeQueue.ts` is the message-persistence wrapper over `writeQueueCore.ts`: it adds message serialization, the `removeIfEmpty` key-removal path, and the `(key, storage)` source identity that flushes a stale pending write when the persistence source changes.
- `readLifecycle.ts` resolves default storage, performs the render-time initial read (`initializePersistenceState`), and keeps the `usePersistenceReadLifecycle` effect as IO/cancellation glue. Preserve write-version gates before applying any async result. It re-exports the planning types/functions so `useChorusPersistence` still imports them from this path.
- `readPlanning.ts` holds the pure `planReadSourceChange`/`planReadSuccess`/`planReadFailure` planners, their plan types (`ReadSourcePlan`/`ReadSuccessPlan`/`ReadFailurePlan`), and the `pendingPreloadMatchesSource`/`shouldClearPendingPreload`/`shouldApplyRead` predicates — no React, no mutation.
- `readApply.ts` holds the imperative `applyRead*` functions and the `ReadLifecycleEffectAdapter` type that translate plans into ref mutations / setState calls.
- `preloadReplay.ts` is the audit point for pre-load `onChange` replay planning: replay only when the read succeeds and `getItem()` returned exactly `null`; stored empty/corrupt strings must not be clobbered.
- `localStorageSync.ts` is only for default `window.localStorage` cross-tab `storage` events; custom/async adapters are responsible for their own notifications. External events are queued behind any in-flight local write (via `writeCoordination`) and rebased against the freshest `stateRef.current` snapshot before applying, so an external value cannot clobber a write that is still settling. An armed-but-not-started debounced write is dropped (`dropPendingWrite()`) only once the external value is confirmed applicable — inside `applyExternalValue` after `stateFromRaw` succeeds and before `setState`, never in `processExternalValue` ahead of the parse. A corrupt external payload hits the `stateFromRaw` early-return and is never applied; dropping the pending write before that point would discard an un-persisted local change (lost update) and desync in-memory state from storage, so the drop is deferred until the value is known good.
